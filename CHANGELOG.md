# Changelog

## 1.1.0 ŌĆö fixed-count OCCURS

- Fixed-count `OCCURS` is now a supported feature rather than a blanket rejection. An `OCCURS n TIMES` item (`TIMES` optional) ŌĆö an elementary field or a whole group ŌĆö is expanded into `n` 1-indexed leaf fields with sequentially computed offsets, one full iteration's worth of bytes apart. This keeps the flat `{Name, Start, Len, Type, ŌĆ”}` model every consumer (`FlatFileCodec`, the JSON contract, `OutSystemsPreview`, the Integration Studio actions) already runs on, rather than introducing a parallel nested/array shape.
- Naming convention: elementary `OCCURS` yields `NAME(1)`ŌĆ”`NAME(n)`; a group `OCCURS` propagates the parenthesized index into every descendant, e.g. `LINE-ITEM(2)-ITEM-QTY`. The `-` separator between the indexed parent and a child is unambiguous because `(` and `)` are illegal in a COBOL identifier ŌĆö so an expanded name can never collide with a real field name. Matches COBOL's own 1-indexed table convention.
- `INDEXED BY name` and `ASCENDING`/`DESCENDING KEY IS name` sub-clauses on the same statement are tolerated and skipped (index-access concerns PICASSO doesn't model), without disturbing the repeat count or any following clause.
- `OCCURS ŌĆ” DEPENDING ON` (variable-length / ODO) stays rejected, now with its own distinct, named error separate from the fixed-count path ŌĆö the `m TO n` bounds are never mistaken for a fixed count. Its record length is data-dependent at decode time, which PICASSO's compute-once offset model can't express; out of scope by design.
- Nested `OCCURS` (a table of tables) is rejected with its own distinct, named error ŌĆö a deliberate, documented non-goal for this pass, separate from ODO.
- The prior blanket `OCCURS` rejection (added alongside `REDEFINES`) is replaced for the fixed-count case; `REDEFINES` handling is unchanged and still rejected.

## 1.0.0 ŌĆö initial release

- Copybook parser: multi-level group/elementary structures, `PIC 9(n)`, `PIC X(n)`, implied decimal (`V`), `SIGN IS LEADING/TRAILING SEPARATE`, and COMP-3 packed decimal.
- Headless copy members: a copybook with no `01` of its own ŌĆö entries starting at 03 or 05, written to be `COPY`'d into a record the including program names ŌĆö gets one supplied, named via the optional `recordName` parameter and flagged by `ParsedCopybook.RootIsSynthetic`. The wrapper is structural: no field's offset or length changes. Two `01`-level records in one copybook remains an error, and is now a distinct one ŌĆö those are alternative layouts, not fields of one record.
- Both COBOL source formats: free-format (`*>` comments) and traditional fixed-format ŌĆö columns 1ŌĆō6 sequence numbers stripped, a column-7 `*` treated as a comment line, columns 73ŌĆō80 (identification area) ignored. Detected per line, so a file can mix both.
- Generic fixed-width codec, decoding/encoding against the derived layout.
- Undelimited fixed-length record files (the mainframe's RECFM=F): records sliced by the layout's own width, with no delimiters between them. Opt-in via `RecordFormat.FixedLength`, or `RecordFormat = "FIXED"` on the `DecodeRecords`/`EncodeRecords` actions; newline-delimited remains the default. Never auto-detected ŌĆö a delimited file's length can divide evenly by the record length too, so guessing would silently yield misaligned records. The fixed-length path does no newline normalization: in an undelimited file 0x0D and 0x0A are data, and DTAR020.bin really does carry 0x0D inside COMP-3 fields.
- EBCDIC cp037 text support, applied per field so COMP-3 bytes are never translated ŌĆö a record-level translation would corrupt every packed-decimal field, the same way an FTP ASCII-mode transfer does. Opt-in via `CharacterEncoding.Ebcdic037`, or `TextEncoding = "EBCDIC"` on the `DecodeRecords`/`EncodeRecords` actions; ASCII/Latin-1 remains the default. The cp037 table is generated from .NET's own code page rather than hand-transcribed, and is pinned against real EBCDIC data.
- OutSystems Structure preview (Integer/Decimal/Text attribute rows) from the same layout.
- Integration Studio action surface (`ParseCopybook`, `DecodeRecords`, `EncodeRecords`, `GetSampleCopybook`, `ListSampleIds`, `DecodeRecordsFromBinary`, `EncodeRecordsToBinary`).
- `Binary Data` variants of `DecodeRecords`/`EncodeRecords`, for a caller whose bytes are already `Binary Data` (an SFTP `Get`, a file upload) rather than `Text` ŌĆö the byteŌåöchar conversion happens inside PICASSO via a new shared `Latin1.cs` helper, rather than requiring the calling app to convert correctly itself. See `docs/mainframe-ingestion.md`.
- 11 bundled sample copybooks (9 real CATALOG-74 layouts, 1 synthetic, 1 genuine 1990s mainframe copybook), each with data attached, embedded in the assembly. None of the data is hand-authored: 7 files are CATALOG-74's own seed data, 2 are real GnuCOBOL output captured by running its batch programs, 1 is generated through PICASSO's own encoder, and 1 is a real mainframe extract, unmodified.
- Samples carry the settings their data needs (`TextEncoding`, `RecordFormat`) on the descriptor, reported by `ListSampleIds` and `GetSampleCopybook` in the exact vocabulary `DecodeRecords` accepts. Neither is detectable from the bytes, so the sample states them rather than the caller guessing: `dtar020-rec` is EBCDIC and undelimited, and every other sample is neither.
- Golden parity test against CATALOG-74's hand-transcribed `specs.js`, byte-for-byte.
- Roundtrip tests (decode ŌåÆ encode ŌåÆ byte-identical) over every bundled sample, including files written by a real COBOL runtime.

Both its files are bundled byte-for-byte as downloaded ŌĆö fixed-format, sequence numbers intact, headless, EBCDIC, undelimited, with nothing adapted for PICASSO ŌĆö and its decoded values match independently-published ones exactly. - `OCCURS` and `REDEFINES` are now rejected with a named error rather than silently mis-parsed. Both used to fall through to the same skip-one-token handling as genuinely irrelevant clauses (`VALUE`, `JUSTIFIED`) ŌĆö but skipping either changes the real byte layout, so a repeating or overlapping field used to parse "successfully" with silently wrong offsets for every field after it. - Not supported yet: `OCCURS`, `REDEFINES` (both now rejected loudly rather than silently mis-parsed), overpunched (non-separate) signed DISPLAY numerics, EBCDIC variants other than cp037, and more than one `01`-level record per copybook.
