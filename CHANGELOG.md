# Changelog

## Unreleased

### Robustness & fail-loud hardening

- **Elementary item with subordinates now fails loudly.** An item with a `PICTURE` that also carried a subordinate (e.g. a `10` under a `05 тАж PIC X`) used to silently drop the subordinate and shift every following offset тАФ a silent miscompute. It is now a named rejection.
- **Unrecognized `COMPUTATIONAL` usage now fails loudly.** A mistyped or unsupported computational usage (`COMPUTATIONAL3` with a dropped hyphen, `COMP-0`, `COMP-7`) used to be silently skipped, leaving the field mis-sized as `DISPLAY`. It is now rejected with a named error. Harmless no-storage clauses (`VALUE`, `JUSTIFIED`, тАж) are still tolerated.
- **Alphanumeric column-1тАУ6 sequence labels are stripped** тАФ real fixed-format copybooks put labels like `JL0001` or `RESERV` in the sequence area, not just six digits; these are now recognized and dropped rather than mis-read as a level number.
- **Off-column `*` comment banners are recognized** тАФ a `*` starting Area A (column 8) with column 7 blank, as several exported copybooks carry, is now treated as a full-line comment instead of gluing onto the first data item.
- **A short / space-padded column-1тАУ6 sequence area is now recognized** тАФ a real fixed-format copybook (royopa/python-cobol `example.cbl`) numbers cols 1-6 with fewer than six digits, e.g. literally `00000 ` (five zeros then a space), or `000010`, or `12   `. The compiler ignores the sequence area's content, so this is legal COBOL (cobc compiles it тЖТ record length 44), but the six-solid-digit and six-solid-character detectors both missed the space, so the sequence area and the cols-73-80 identification tail leaked into the tokenizer and falsely rejected the layout. Such a line is now detected (cols 1-6 digits-and-spaces with at least one digit) and stripped like any fixed-format line. This includes a fixed-format **continuation** line тАФ a clause operand (an `OCCURS` count, a `PIC`) placed on its own line, whose sequence number is stripped only when the previous statement is still open and column 1 is a digit (a left-aligned sequence number); left unstripped a **nonzero** sequence number leaked into the joined statement and was consumed as a real operand, silently mis-sizing the record (e.g. a `130` sequence number on an `OCCURS`-count line built a 130-element table). **Free-format safety is preserved:** the detection keeps the same level-number guard the alphanumeric-sequence path uses тАФ an all-blank area (no digit) is treated as free-format indentation, and a free-format line whose cols 1-6 hold digits only because the level number is indented into them (`    05  B тАж`, "05" at cols 5-6) is not misread, because its code area opens with the DATA-NAME rather than a level number. When the signals are ambiguous the line is left as free-format (fail loud) rather than truncated. (`tools/layout-oracle` mirrors the same handling тАФ independently тАФ so it stays at parity with the parser and can validate this class against GnuCOBOL.)
- **A storageless item now fails loudly.** A pic-less, childless item (`05 B.` тАФ no `PICTURE`, no width-bearing `USAGE`, no subordinates) used to be dropped silently, sliding every following field forward and under-sizing the record; it is now a named rejection (a real compiler rejects the same input). Level-78 named constants are tolerated and dropped (they carry no storage, like level 88), so a copybook mixing 78 constants with data fields still parses; `PROGRAM-POINTER` joins the rejected pointer family with its own named error.
- **The optional word `IS` after `PIC`/`PICTURE` is now tolerated.** `PICTURE IS X(9)` / `PIC IS 99` тАФ standard COBOL, where `IS` is an optional noise word тАФ used to fail with "Unrecognized character in PIC clause". The keyword is now skipped before the picture string, the same way the `SIGN IS ...` clause already tolerates it.
- **The fixed-format identification area (cols 73-80) may hold text, not just a sequence number.** With columns 1-6 blank, PICASSO used to strip cols 73-80 only when they were all digits/spaces тАФ so a real copybook that annotates the area (e.g. a `POSITION` header and per-field byte offsets, as the Texas RRC well-data layout does) leaked the text into the tokenizer and was falsely rejected. It's now also stripped when the code area (cols 1-72) already completes a statement (balanced quotes, ends with `.`), which a compiler-ignored identification area always follows. Free-format source is safe: a line whose real content (a `USAGE` clause, a long literal) runs past column 72 has no terminating `.` there, so it is left intact, never truncated. (`tools/layout-oracle` mirrors the same rule so it stays at parity with the parser.)
- **A blank / non-numeric DISPLAY numeric now fails loudly with the field named.** Decoding a space-filled or otherwise non-digit `DISPLAY` numeric field used to surface a bare, context-free .NET `FormatException`; it now throws a named error identifying the field and its bytes. A blank numeric is deliberately **not** auto-zeroed тАФ that would be a silent guess about intent; the mismatch is surfaced instead so the layout/offset/encoding can be checked.
- **`OCCURS` on a level-01 or level-77 item is now rejected.** OCCURS is illegal on a record (01) or a standalone item (77) тАФ neither may repeat. PICASSO used to silently expand it (`01 X PIC 9(3) OCCURS 4` became a 12-byte record), disagreeing with a real compiler; it is now a named rejection (fixed-count and `DEPENDING ON` alike). A repeating item must sit under a group.
- **A REDEFINES overlay inside an `OCCURS тАж DEPENDING ON` record no longer corrupts.** The ODO codec paths now measure the record with the overlap-aware `RecordLength` (max start+len) and reject an overlapping *encode* тАФ matching the fixed-layout REDEFINES policy (decode supported, encode refused). Previously encode silently emitted a too-long record (both overlay readings written) and decode rejected the valid record on a summed length.

### Nested fixed-count `OCCURS` (table of tables)

- Nested **fixed-count** `OCCURS` тАФ an `OCCURS` item inside another `OCCURS`, all fixed counts, to arbitrary depth тАФ is now supported (was a named rejection). It flattens **recursively**: the outer table expands to N copies, each holding M copies of the inner (and deeper), with sequential offsets. Every level's 1-based index rides in the leaf name, one `(index)` segment per `OCCURS` level, matching the single-level style тАФ e.g. `LINE OCCURS 3` over `ITEM OCCURS 4` over `CODE PIC X(2)` gives `LINE(1)-ITEM(1)-CODE`@0 тАж `LINE(3)-ITEM(4)-CODE`@22, a 3├Ч4├Ч2 = 24-byte record. The outer element's stride is the full byte size of one fully-expanded inner element.
- Implementation: `ComputeOffsets` already recursed and multiplied correctly (nested offset math was never the gap). The flattening pass was generalized тАФ `Flatten`/`EmitIndexedLeaves` are replaced by a single compositional `EmitSubtree` that carries an accumulated byte-shift and name-prefix, so an OCCURS level expands into indexed copies and recurses, and an inner OCCURS expands in turn. `RejectNestedOccurs` (which only ever caught fixed-in-fixed nesting) is removed.
- The checked overflow guard in `ComputeOffsets` multiplies across every nesting level, so a pathological product (e.g. `OCCURS 100000` inside `OCCURS 100000`) fails loud rather than wrapping.
- Still rejected, unchanged, each with its own named error: an `OCCURS тАж DEPENDING ON` table nested inside another `OCCURS`, and any `OCCURS` (fixed or ODO) nested inside an ODO table тАФ only *fixed*-count nesting is in scope; any variable-length nesting stays gated (enforced by `ValidateAndFindOdos`).
- Validated against GnuCOBOL (`tools/layout-oracle`, `cobc -std=mvs`): 4/4 nested fixtures (2-level, 3-level, mixed group + trailing field, and COMP-3-in-nested) agree exactly, 0 disagreements. Single-level `OCCURS` output is byte-identical (dedicated regression test).

### Named binary usages (`BINARY-CHAR`/`SHORT`/`LONG`/`DOUBLE`)

- The fixed-width named binary usages тАФ `BINARY-CHAR` (1 byte), `BINARY-SHORT` (2), `BINARY-LONG` (4), `BINARY-DOUBLE` (8), with optional `SIGNED` (default) / `UNSIGNED` тАФ are now **supported**. Unlike `COMP`/`BINARY`, they carry no `PIC`: the width comes from the usage name, and the value decodes big-endian, two's-complement (signed) or plain magnitude (unsigned). Signedness affects the value, not the width тАФ a 1-byte `0xFF` is `-1` signed, `255` unsigned; a 4-byte `0xFFFFFFFF` is `-1` vs `4294967295`.
- This closed a **silent miscompute**: a pic-less named-binary item used to build no `FieldSpec`, so it was treated as a childless group and dropped тАФ every following field slid forward and the record was under-sized, with no error. (`01 R. 05 A PIC X(3). 05 B USAGE BINARY-LONG. 05 C USAGE BINARY-SHORT. 05 D PIC X(3).` is 12 bytes to GnuCOBOL; PICASSO used to report 6.)
- A `PIC` clause combined with a named binary usage (the fixed width would contradict the digit count), and GnuCOBOL's platform-dependent `BINARY-C-LONG` (no standard width), are both **rejected loudly** rather than guessed. `SIGN IS тАж SEPARATE` on a named binary is refused (its sign is intrinsic to two's complement), as it already is for `COMP`.
- Implementation: `Binary` gains a width-explicit path (`Decode(data, scale, signed)` / `EncodeWidth(value, width, scale, signed)`) driven by the byte width, since the digit-derived `ByteLength` cannot yield 1 byte. A named-binary `FieldSpec` sets `Digits = 0` to mark it width-explicit; the codec routes `Digits == 0` through the width path and the existing digit-derived `COMP`/`COMP-4`/`BINARY` path is unchanged.
- Validated against GnuCOBOL (`tools/layout-oracle`, `cobc -std=mvs`): the minimal fixture (12) and the DB2 **`SQLCA`** copybook (157 тАФ two `BINARY-LONG` counters, a `BINARY-SHORT` length, a `BINARY-LONG OCCURS 6` array, a `REDEFINES`, DISPLAY text) both agree exactly. SQLCA is `COPY`'d into essentially every DB2 program.

### `OCCURS 0 TO n`; multiple flat ODO tables per record

- `OCCURS тАж DEPENDING ON` now accepts a **lower bound of 0** (`OCCURS 0 TO n`, a table that may be empty): a resolved count of 0 makes the table contribute no fields, with everything after it shifting to the table's start.
- **More than one flat, top-level ODO table per record is now supported**, resolved **left-to-right**. A later table's depending field sits at an offset that itself depends on the *earlier* tables' counts (an earlier count of 0 or n shifts it), so the counts cannot be read all at once: for each table the codec builds the partial concrete layout with the tables resolved so far, reads that table's depending field at its now-pinned offset, validates `m тЙд count тЙд n` (out-of-range fails loudly, per table), and continues тАФ then decodes/encodes against the fully-resolved layout. This is a decode-and-encode capability of the engine's `FlatFileCodec` `ParsedCopybook` overloads; `ParsedCopybook.Odos` lists one `OdoInfo` per table in source order and `BuildConcreteLayout(IReadOnlyList<int> counts)` expands them. Validated against GnuCOBOL (`cobc -x -std=mvs`): the record's `LENGTH OF` matches `BuildConcreteLayout`'s total for two- and three-table records across count combinations, including the count-0-for-an-earlier-table shift, with byte-for-byte round-trips (Latin-1 and EBCDIC, delimited and undelimited).
- Still rejected, each with its own named error: an ODO table nested inside another `OCCURS`, an `OCCURS` (fixed or ODO) nested inside an ODO table, and a depending field defined after its table or absent. The Integration Studio JSON action surface still rejects any ODO copybook loudly (its flat spec is static).

### Decode coverage (shipped since 1.1.0)

These landed incrementally after 1.1.0 and are grouped here for the next release. Each is documented in the README's "What it does" and validated against GnuCOBOL (`tools/layout-oracle`, `cobc -std=mvs`); several closed a silent-miscompute where the construct used to be skipped and the record mis-sized.

- **`REDEFINES`** (byte overlay) тАФ a redefining item shares its target's bytes; each overlapping field decodes independently. Encode of an overlapping layout is rejected loudly (never last-write-wins). The nameless form (`05 REDEFINES A тАж`) is handled as a `FILLER` overlay, not laid out as new storage. Supersedes the earlier "REDEFINES still rejected" note in 1.1.0.
- **`SYNCHRONIZED`/`SYNC` alignment** тАФ a binary `COMP`/`BINARY` item marked SYNC is aligned to its 2/4/8-byte boundary via synthetic slack; no-op on non-binary; SYNC in/around an `OCCURS` is rejected loudly.
- **Binary `COMP` integers** тАФ `COMP`/`COMP-4`/`COMP-5`/`BINARY` (big-endian, width by digit count the IBM way), with `PACKED-DECIMAL` aliased to COMP-3.
- **Edited `PIC` pictures** (`Z * . , / B 0 + - $ CR DB`) тАФ measured to their exact display width and surfaced as `Text` (byte passthrough); the numeric value is not re-interpreted.
- **Overpunched (zoned) signs** тАФ `PIC S9(n)` without `SEPARATE`, trailing or leading, the sign folded into a digit's zone nibble.
- **Group-level `USAGE` inheritance** тАФ a `USAGE` on a group item applies to subordinate elementary items that state none of their own (nearest-ancestor wins); closed a silent miscompute where a group `COMP-3`/`COMP` was ignored.
- **Level-88 condition-names** тАФ tolerated and ignored (zero storage).
- **Multiple `01` records** тАФ supported additively via `CopybookParser.ParseRecords` (each an independent layout); single `Parse` still rejects them.
- **`OCCURS тАж DEPENDING ON`** (single flat table) тАФ decode/encode a variable-length record from the depending field's value; nested ODO / OCCURS-in-ODO remain rejected. (Extended above to multiple flat tables per record.)

## 1.1.0 тАФ fixed-count OCCURS

- Fixed-count `OCCURS` is now a supported feature rather than a blanket rejection. An `OCCURS n TIMES` item (`TIMES` optional) тАФ an elementary field or a whole group тАФ is expanded into `n` 1-indexed leaf fields with sequentially computed offsets, one full iteration's worth of bytes apart. This keeps the flat `{Name, Start, Len, Type, тАж}` model every consumer (`FlatFileCodec`, the JSON contract, `OutSystemsPreview`, the Integration Studio actions) already runs on, rather than introducing a parallel nested/array shape.
- Naming convention: elementary `OCCURS` yields `NAME(1)`тАж`NAME(n)`; a group `OCCURS` propagates the parenthesized index into every descendant, e.g. `LINE-ITEM(2)-ITEM-QTY`. The `-` separator between the indexed parent and a child is unambiguous because `(` and `)` are illegal in a COBOL identifier тАФ so an expanded name can never collide with a real field name. Matches COBOL's own 1-indexed table convention.
- `INDEXED BY name` and `ASCENDING`/`DESCENDING KEY IS name` sub-clauses on the same statement are tolerated and skipped (index-access concerns PICASSO doesn't model), without disturbing the repeat count or any following clause.
- `OCCURS тАж DEPENDING ON` (variable-length / ODO) stays rejected, now with its own distinct, named error separate from the fixed-count path тАФ the `m TO n` bounds are never mistaken for a fixed count. Its record length is data-dependent at decode time, which PICASSO's compute-once offset model can't express; out of scope by design.
- Nested `OCCURS` (a table of tables) is rejected with its own distinct, named error тАФ a deliberate, documented non-goal for this pass, separate from ODO.
- The prior blanket `OCCURS` rejection (added alongside `REDEFINES`) is replaced for the fixed-count case; `REDEFINES` handling is unchanged and still rejected.

## 1.0.0 тАФ initial release

- Copybook parser: multi-level group/elementary structures, `PIC 9(n)`, `PIC X(n)`, implied decimal (`V`), `SIGN IS LEADING/TRAILING SEPARATE`, and COMP-3 packed decimal.
- Headless copy members: a copybook with no `01` of its own тАФ entries starting at 03 or 05, written to be `COPY`'d into a record the including program names тАФ gets one supplied, named via the optional `recordName` parameter and flagged by `ParsedCopybook.RootIsSynthetic`. The wrapper is structural: no field's offset or length changes. Two `01`-level records in one copybook remains an error, and is now a distinct one тАФ those are alternative layouts, not fields of one record.
- Both COBOL source formats: free-format (`*>` comments) and traditional fixed-format тАФ columns 1тАУ6 sequence numbers stripped, a column-7 `*` treated as a comment line, columns 73тАУ80 (identification area) ignored. Detected per line, so a file can mix both.
- Generic fixed-width codec, decoding/encoding against the derived layout.
- Undelimited fixed-length record files (the mainframe's RECFM=F): records sliced by the layout's own width, with no delimiters between them. Opt-in via `RecordFormat.FixedLength`, or `RecordFormat = "FIXED"` on the `DecodeRecords`/`EncodeRecords` actions; newline-delimited remains the default. Never auto-detected тАФ a delimited file's length can divide evenly by the record length too, so guessing would silently yield misaligned records. The fixed-length path does no newline normalization: in an undelimited file 0x0D and 0x0A are data, and DTAR020.bin really does carry 0x0D inside COMP-3 fields.
- EBCDIC cp037 text support, applied per field so COMP-3 bytes are never translated тАФ a record-level translation would corrupt every packed-decimal field, the same way an FTP ASCII-mode transfer does. Opt-in via `CharacterEncoding.Ebcdic037`, or `TextEncoding = "EBCDIC"` on the `DecodeRecords`/`EncodeRecords` actions; ASCII/Latin-1 remains the default. The cp037 table is generated from .NET's own code page rather than hand-transcribed, and is pinned against real EBCDIC data.
- OutSystems Structure preview (Integer/Decimal/Text attribute rows) from the same layout.
- Integration Studio action surface (`ParseCopybook`, `DecodeRecords`, `EncodeRecords`, `GetSampleCopybook`, `ListSampleIds`, `DecodeRecordsFromBinary`, `EncodeRecordsToBinary`).
- `Binary Data` variants of `DecodeRecords`/`EncodeRecords`, for a caller whose bytes are already `Binary Data` (an SFTP `Get`, a file upload) rather than `Text` тАФ the byteтЖФchar conversion happens inside PICASSO via a new shared `Latin1.cs` helper, rather than requiring the calling app to convert correctly itself. See `docs/mainframe-ingestion.md`.
- 11 bundled sample copybooks (9 real CATALOG-74 layouts, 1 synthetic, 1 genuine 1990s mainframe copybook), each with data attached, embedded in the assembly. None of the data is hand-authored: 7 files are CATALOG-74's own seed data, 2 are real GnuCOBOL output captured by running its batch programs, 1 is generated through PICASSO's own encoder, and 1 is a real mainframe extract, unmodified.
- Samples carry the settings their data needs (`TextEncoding`, `RecordFormat`) on the descriptor, reported by `ListSampleIds` and `GetSampleCopybook` in the exact vocabulary `DecodeRecords` accepts. Neither is detectable from the bytes, so the sample states them rather than the caller guessing: `dtar020-rec` is EBCDIC and undelimited, and every other sample is neither.
- Golden parity test against CATALOG-74's hand-transcribed `specs.js`, byte-for-byte.
- Roundtrip tests (decode тЖТ encode тЖТ byte-identical) over every bundled sample, including files written by a real COBOL runtime.
- Real-world validation against DTAR020, a genuine mainframe copybook and its real 379-record extract (not written for this project, sourced from github.com/bmTas/CobolToJson). Both its files are bundled byte-for-byte as downloaded тАФ fixed-format, sequence numbers intact, headless, EBCDIC, undelimited, with nothing adapted for PICASSO тАФ and its decoded values match independently-published ones exactly. The entire 10,233-byte file тАФ fixed-format copybook, EBCDIC text, packed decimals, no record delimiters тАФ round-trips through the public API to the same bytes it arrived as.
- `OCCURS` and `REDEFINES` are now rejected with a named error rather than silently mis-parsed. Both used to fall through to the same skip-one-token handling as genuinely irrelevant clauses (`VALUE`, `JUSTIFIED`) тАФ but skipping either changes the real byte layout, so a repeating or overlapping field used to parse "successfully" with silently wrong offsets for every field after it. Fixed as a straightforward extension of the level-66/88 rejection pattern already in place.
- Not supported yet: `OCCURS`, `REDEFINES` (both now rejected loudly rather than silently mis-parsed), overpunched (non-separate) signed DISPLAY numerics, EBCDIC variants other than cp037, and more than one `01`-level record per copybook.
